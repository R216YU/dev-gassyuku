# インフラ構築検討：VPCおよびLambda周辺設計

これまでの対話に基づき、コスト効率とセキュリティ、将来の拡張性を両立したインフラ構成の検討結果をまとめます。

## 1. 当初の要望と課題
- **要望**: CloudFormationを使用してVPCを作成したい。コストを抑えるため、高額なNAT Gatewayを避け、パブリック構成にしたい。
- **課題**: 
    - **パブリックIPv4アドレスの有料化**: 現在、パブリックIPを持つだけで月額約500円強のコストがかかる。
    - **Lambdaの制約**: LambdaをVPC内のパブリックサブネットに置いても、パブリックIPを持てないためインターネット（Bedrock API）には出られない。

## 2. コスト比較（月額目安）

| 構成案 | 通信手段 | 推定固定費 | 備考 |
| :--- | :--- | :--- | :--- |
| **案1: NAT Gateway構成** | NAT Gateway | 約5,000円〜 | 最も汎用的だが高コスト |
| **案2: パブリック構成** | IGW + パブリックIP | 約500円〜/IP | EC2なら可。Lambdaは不可。 |
| **案3: VPC Endpoint構成** | Interface Endpoint | 約1,000円〜/個 | **本案採用（Bedrock用途）** |
| **案4: VPCなし構成** | 直接インターネット | **0円** | セキュリティ要件が許せば最強 |

## 3. 推奨アーキテクチャ設計
将来的にRDSやElastiCacheなどのデータベース系サービスをVPC内に配置することを想定し、以下の構成で構築を進めます。

### ネットワーク構成
- **VPC / Subnets**: 
    - 将来の拡張（Load Balancer等）のための「パブリックサブネット（2 AZ）」。
    - Lambda, RDS, ElastiCacheを隠すための「プライベートサブネット（2 AZ）」。
- **Gateway**: 
    - パブリックサブネット用の「Internet Gateway」。
    - S3へのアクセスを無料・高速化する「S3 Gateway Endpoint」。

### Bedrockアクセス（コスト最適化）
- **Interface VPC Endpoint**:
    - NAT Gateway（5,000円）の代わりに、Bedrock Runtime用のエンドポイント（1,000円強）を設置。
    - これにより、Lambdaを安全なプライベートサブネットに置いたまま、Bedrock APIを利用可能にする。

## 4. テンプレート比較：VPC Endpoint版 vs NAT Gateway版

2つの構成パターンのCloudFormationテンプレートを作成し、比較検討を行いました。

### 作成したテンプレート
| ファイル名 | 方式 |
| :--- | :--- |
| `vpc-serverless_vpc-endpoint.yaml` | VPC Endpoint方式（Bedrock・S3専用通路） |
| `vpc-serverless_nat-gateway.yaml` | NAT Gateway方式（インターネット全体への代理通信） |

### 共通リソース（両テンプレートで同じ）
- VPC（10.0.0.0/16）
- Internet Gateway
- パブリックサブネット x2（ALB・NAT Gateway等の配置用）
- プライベートサブネット x2（Lambda・RDS等の配置用）
- パブリックルートテーブル（IGWへのルート）
- Lambda用セキュリティグループ

### 差分リソース

#### VPC Endpoint版のみに存在するリソース
| リソース | 説明 | 料金 |
| :--- | :--- | :--- |
| S3 Endpoint (Gateway型) | S3専用の通路 | **無料** |
| Bedrock Runtime Endpoint (Interface型) | Bedrock専用の通路 | 約1,000〜2,000円/月 |
| Endpoint用セキュリティグループ | Lambdaからの443ポートのみ許可 | 無料 |

#### NAT Gateway版のみに存在するリソース
| リソース | 説明 | 料金 |
| :--- | :--- | :--- |
| Elastic IP | NAT Gateway用の固定パブリックIP | 約500円/月 |
| NAT Gateway | パブリックサブネットに配置する代理通信装置 | 約4,500円/月 |
| プライベートデフォルトルート | 0.0.0.0/0 → NAT Gatewayへのルート | 無料 |

### 通信の流れの違い
```
【VPC Endpoint版】
Lambda → (VPC内部) → Bedrock Endpoint → Bedrock  ✅
Lambda → (VPC内部) → S3 Endpoint → S3             ✅
Lambda → Slack等の外部API                          ❌ 通信不可

【NAT Gateway版】
Lambda → NAT Gateway → IGW → インターネット → 何でもOK  ✅
```

### 総合比較
| 観点 | VPC Endpoint版 | NAT Gateway版 |
| :--- | :--- | :--- |
| **月額固定費** | 約1,000〜2,000円 | **約5,000円〜** |
| **データ転送料金** | なし（AWS内部通信） | $0.045/GB（NAT経由） |
| **通信先の自由度** | Bedrock・S3のみ | **何でもOK（インターネット全体）** |
| **セキュリティ** | **◎ 最も堅牢**（外への道がない） | ○ 良好（SGで制御） |
| **設定の手間** | サービスごとにEndpointが必要 | **一度設定すれば全通信OK** |

### AZ数と料金の関係（重要）
VPC EndpointもNAT Gatewayも、**配置するAZの数に比例して料金が増加**する。
これは両方とも、AZごとに専用のネットワークリソース（ENIやNAT装置）が作成されるためである。

#### VPC Endpoint（Interface型）の場合
指定したサブネット（AZ）ごとにENI（仮想ネットワークカード）が1つ作成され、それぞれに課金される。

| AZ数 | ENI数 | 時間単価 | 月額（東京リージョン） |
| :--- | :--- | :--- | :--- |
| 1AZ | 1 | $0.014/h | 約$10（約1,500円） |
| 2AZ | 2 | $0.028/h | 約$20（約3,000円） |

#### NAT Gatewayの場合
AZごとにNAT Gateway + Elastic IPの組み合わせが必要になる。1AZ構成でも動作するが、そのAZが障害を起こすと全プライベートサブネットがインターネットに出られなくなる。

| AZ数 | NAT GW数 | 時間単価 | 月額（東京リージョン） |
| :--- | :--- | :--- | :--- |
| 1AZ | 1 | $0.062/h | 約$45（約6,800円） |
| 2AZ | 2 | $0.124/h | 約$90（約13,600円） |

#### 推奨
- **開発・検証用途**: 1AZ構成で十分。コストを半額に抑えられる。
- **本番用途**: 2AZ構成にして冗長性を確保する。

> ※ 作成済みテンプレートは両方とも1AZ構成（NAT Gateway版）または2AZ構成（VPC Endpoint版）で作成している。VPC Endpoint版を1AZに変更する場合は `SubnetIds` からサブネットを1つ削除するだけで対応可能。

### 選定基準
- **「Bedrock・S3だけ使えればいい、コストを抑えたい」** → VPC Endpoint版
- **「外部API（Slack通知等）も呼びたい、設定をシンプルにしたい」** → NAT Gateway版

## 6. 最終結論：VPCなし構成を採用

上記の検討を踏まえ、本プロジェクトでは **「VPCなし構成（案4）」** を採用する。

### 採用理由
- **コスト**: ネットワーク関連の固定費が **0円** となり、最もコスト効率が良い。
- **シンプルさ**: VPC、サブネット、ルートテーブル、エンドポイント等のネットワーク設計・管理が不要。
- **用途との整合性**: 現時点ではLambdaからBedrock APIを呼び出すことが主な用途であり、VPC内のリソース（RDS、ElastiCache等）へのアクセスは不要。VPCに入れなくてもLambdaは標準でBedrockにアクセス可能。
- **セキュリティ**: IAMポリシーによるアクセス制御で十分にセキュリティを担保できる。

### 将来の拡張について
RDSやElastiCacheなど、VPC内に配置すべきリソースが必要になった場合は、本ドキュメントで検討した構成（VPC Endpoint版 or NAT Gateway版）を改めて導入する。作成済みのCloudFormationテンプレートをそのまま利用可能。

## 7. 最終方針見直し：最小コストVPC（Private Subnetのみ＋S3 Gateway Endpoint）の採用

当初は「VPCなし構成」を採用するとしていましたが、セキュリティ要件や将来的なリソース（RDS等）の配置に備え、LambdaをVPC内に配置しつつもコストを極限まで抑えた **「1AZ/Private Subnetのみの最小構成VPC」** を構築する方針へ見直しました。

### 採用するテンプレート
- `vpc-serverless_none.yaml`

### 構成のポイント
- **Public Subnet / Internet Gateway なし**: 外部からの直接アクセスを受け付けるリソースが当面不要であるため、パブリック構成を排除し、セキュリティリスクを下げています。
- **Private Subnet のみ (1AZ)**: 無料枠での検証を前提とするため、利用するAvailability Zoneを1つに絞り最小構成としています。
- **S3 Gateway Endpoint (無料) の配置**: LambdaをVPC内のPrivate Subnetに配置した場合でも、S3への通信（ファイルの配置等）を可能にするため、完全無料の Gateway型 Endpoint のみ設置しています。
- **有料エンドポイントの排除**: NAT GatewayやBedrock等の有料 Endpoint を全て削除しているため、ネットワークに関する固定費は実質 **0円** となります。
